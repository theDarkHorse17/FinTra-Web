# Testing Guide

This page explains how to test the FinTra application across devices, builds, and environments. It covers manual testing, automated testing (future), debugging steps, and guidelines for internal testers.

## 1. Testing Overview

FinTra is currently in **MVP stage**, so testing focuses primarily on:

- App stability
- Expense logging accuracy
- UI/UX consistency
- Month-key calculations
- Database correctness
- Crash-free navigation
- Edge-case handling

Testing ensures every user action behaves predictably and securely.

## 2. Types of Testing

### Manual Testing (Current Focus)
All testers validate:
- UI flows
- Expense addition
- Savings updates
- Category-level breakdown
- Login & logout
- Month switching
- Overview calculations

### Automated Testing (Phase 2)
Planned for future:
- Unit tests
- Integration tests
- Widget tests

## 6. Bug Reporting Process

When reporting a bug, include:

### 1. Steps to reproduce
Clear step-by-step actions.

### 2. Expected result
What should happen.

### 3. Actual result
What actually happened.

### 4. Device details
- Device model
- Android version
- App version

### 5. Screenshot / Screen recording
Very helpful for debugging.

**Example:**

## 7. Crash Testing

Try to intentionally break the app:

- Rapid tapping on buttons
- Switching tabs continuously
- Repeatedly adding/removing expenses
- Opening dialogs and rotating the phone
- Killing and reopening app quickly

Any crash must be reported.

## 8. Performance Testing

Focus on:

- Smooth scrolling
- Zero jank in page transitions
- Expense addition < 300ms
- Savings updates < 200ms
- Overview load time < 1 sec

Use the Flutter Performance Overlay (optional).

## 9. Pre-release Checklist

Before pushing a new APK:

- [ ] All pages load without errors
- [ ] No crashes in navigation
- [ ] Expenses update instantly
- [ ] Overview shows correct month data
- [ ] Savings updates correctly
- [ ] No debug logs in release build
- [ ] App icon + version updated
- [ ] Obfuscation build tested
